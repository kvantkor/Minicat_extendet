boot.asm
Список модулей инициализации (Boot Sequence)
Multiboot Header: Точка входа для GRUB/QEMU. Передает информацию о памяти и загруженных модулях.
GDT (Global Descriptor Table): Установка плоской модели памяти (0-4 ГБ). Переход в защищенный режим с чистыми сегментными регистрами.
PIC Remap: Перепрограммирование контроллеров прерываний 8259. Сдвигает аппаратные прерывания (IRQ) выше программных исключений процессора.
IDT (Interrupt Descriptor Table): Инициализация таблицы векторов прерываний. Регистрация обработчиков исключений, таймера, клавиатуры и системных вызовов.
PIT (Programmable Interval Timer): Настройка системного таймера на частоту 100 Гц для планировщика задач.
Interrupt Stub (Common Stub): Универсальный шлюз, который сохраняет состояние CPU (pushad) и передает управление в высокоуровневый C-код (exception_handler).
Структурированный список функций
Я разделил функции по их назначению. Ты можешь использовать этот список в README или docs/boot.md.
1. Управление процессором и системные функции
_start: Точка входа (Entry point). Настраивает стек, GDT, IDT и делает прыжок в ядро.
remap_pic: Настраивает связку Master/Slave контроллеров прерываний.
setup_idt: Заполняет таблицу IDT векторами (обработчиками).
setup_timer: Конфигурирует микросхему 8254 (PIT) на нужную частоту.
2. Интерфейс ввода-вывода (I/O)
Эти функции доступны для вызова из C через extern.
outb / inb: Запись/чтение 8-битного байта в порт.
outw / inw: Запись/чтение 16-битного слова в порт.
outl / inl: Запись/чтение 32-битного двойного слова в порт.
io_wait: Короткая задержка для синхронизации работы со старым оборудованием.
3. Обработчики прерываний (Interrupt Service Routines)
isr0: Исключение "Деление на ноль".
irq0: Системный таймер.
irq1: Клавиатура (ввод пользователя).
irq14 / irq15: Контроллеры жестких дисков (ATA Primary/Secondary).
isr_syscall: Программное прерывание int 0x30 для системных вызовов из User Mode.
common_stub: "Клей" между ассемблером и C. Сохраняет регистры и вызывает exception_handler.
Глобальные переменные (Экспортируемые)
boot_magic: Магическое число Multiboot (подтверждение корректной загрузки).
boot_mboot_ptr: Указатель на структуру с данными о железе от загрузчика.
idt_table: Адрес самой таблицы прерываний в памяти.



1. Проверка окружения (Bootstrapping)
Валидация Multiboot: Ядро проверяет значение в регистре eax (переданное через boot_magic). Если оно не равно 0x2BADB002, система останавливается. Это гарантирует, что загрузчик (например, GRUB) корректно передал данные.
Определение памяти: Из структуры multiboot_info извлекается объем доступной оперативной памяти.
Если установлен флаг 0x1, объем вычисляется как mem_upper * 1024 + 1MB.
Если данные недоступны, устанавливается безопасный лимит в 16 МБ.
2. Управление памятью (Memory Management)
PMM & VMM: Вызываются pmm_init (физический менеджер) и vmm_init (виртуальный менеджер).
Проверка пейджинга: Код считывает регистр CR0. Если 31-й бит установлен, выводится подтверждение, что страничная адресация активна.
3. Многозадачность (Multitasking)
Инициализация ядра как задачи: Создается структура task_t для текущего потока выполнения (kernel task). Ему присваивается PID, текущий каталог страниц и имя "kernel". Список задач зацикливается на самого себя (current_task->next = current_task).
Создание системных задач: С помощью create_task запускаются фоновые процессы:
Status_os_bar: вероятно, отрисовка интерфейса.
run_CAT: выполнение пользовательской или системной утилиты.
cleaner: "Жнец" (Kernel Cleaner) для освобождения ресурсов завершенных задач.
4. Файловая система и ввод-вывод
FAT32: Инициализация драйвера файловой системы.
Shell: Запуск командной оболочки для взаимодействия с пользователем.
Прерывания: Команда asm volatile("sti") разрешает аппаратные прерывания, что фактически запускает планировщик задач (переключение контекста обычно происходит по прерыванию таймера).
5. Режим ожидания
После инициализации ядро уходит в бесконечный цикл с инструкцией hlt. Это переводит процессор в состояние низкого энергопотребления до следующего прерывания.

3. Системные и демонстрационные задачи
В ядре реализованы два типа задач: фоновые (сервисные) и демонстрационные (визуальные).
3.1 Визуальная индикация (Задачи-индикаторы)
Задачи task2_test и task_a служат для визуальной проверки работоспособности планировщика (Multitasking).
Механизм: Напрямую пишут в видеопамять VGA (0xB8000).
task2_test: Отображает вращающийся символ (|/-\) в позиции 79 (правый верхний угол).
task_a: Выводит анимированную надпись ("CAT" / "RUN") в позициях 76-78.
Особенности: Используют volatile циклы задержки для контроля скорости анимации, не блокируя при этом всё ядро благодаря переключению контекста по таймеру.
4. Жизненный цикл задачи и подсистема очистки
4.1 Поиск процессов (find_task)
Функция итерирует по кольцевому списку процессов, начиная с current_task, пока не найдет задачу с нужным PID или не вернется в начало списка. Это гарантирует безопасный обход всех активных задач.
4.2 Механизм освобождения ресурсов (task_cleanup)
Когда задача завершается (переходит в dead_list), вызывается эта функция. Она выполняет:
Очистку VGA-ресурсов: Проходит по списку структур resource_t. Если задача владела ячейкой экрана (тип RES_VGA_SYMBOL), в эту позицию записывается пробел (0x0720).
Освобождение памяти:
Проверяет поле steck_base. Если оно не равно 0 (что отличает пользовательскую задачу от ядра), вызывается kfree_page для возврата физической памяти стека.
4.3 Задача-Жнец (kernel_cleaner_task)
Это системный поток с низким приоритетом, который занимается окончательной утилизацией памяти «зомби-процессов»:
Атомарность: Извлечение задачи из dead_list происходит при выключенных прерываниях (cli/sti), чтобы избежать конфликтов с планировщиком.
Работа в фоне: После каждой итерации проверки списка вызывается пауза, чтобы не занимать процессорное время, необходимое Shell или пользовательским задачам.
Резюме по структуре ресурсов
Поле	Описание
dead_list	Односвязный список задач, ожидающих удаления.
resource_t	Связный список объектов (память, порты, ячейки VGA), закрепленных за задачей.
steck_base	Базовый адрес страницы стека; используется Жнецом для деаллокации.

5. Структуры данных и системные определения
5.1 Управление задачами (task_t)
Структура task_t определяет контекст выполнения процесса. Использование атрибута __attribute__((packed)) гарантирует отсутствие неявных отступов между полями.
Идентификация: id (PID) и name (до 20 символов).
Контекст памяти:
esp: Текущий указатель стека задачи (сохраняется при переключении).
steck_base: Базовый адрес стека (нужен для корректного удаления через kfree_page).
page_directory: Физический адрес каталога страниц (загружается в CR3).
Состояние (stat):
TASK_STATE_RUNNING (0): Активна.
TASK_STATE_SLEEP (1): Ожидание (зарезервировано).
TASK_STATE_KILLED (2): Помечена для удаления Жнецом.
Безопасность: Поле magic в конце структуры служит маркером для детектирования переполнения стека (Stack Overflow) или повреждения памяти.
5.2 Управление ресурсами (resource_t)
Система отслеживает владение ресурсами через связный список, прикрепленный к каждой задаче.
Типы ресурсов (res_type_t):
RES_VGA_SYMBOL: Символ или область в видеопамяти.
RES_MEMORY: Выделенный блок памяти.
Механизм: Поля addr и value позволяют Жнецу (cleaner) точно знать, что именно нужно освободить или «затереть» при завершении задачи.
5.3 Состояние процессора (registers_t)
Эта структура строго соответствует порядку пуша регистров в стек обработчиком прерываний (в boot.asm).
Сегменты: ds, es, fs, gs.
Регистры общего назначения: Сохраненные инструкцией pushad.
Метаданные прерывания: Номер прерывания (int_no) и код ошибки.
Контекст IRET: eip, cs, eflags — пушатся самим процессором при возникновении прерывания.
6. Низкоуровневый I/O (Ассемблерные обертки)
Ядро взаимодействует с оборудованием через функции, реализованные в ASM и объявленные как extern:
Функция	Описание
outb / inb	Запись / Чтение 8-битного байта из порта (клавиатура, PIC).
outw / inw	Запись / Чтение 16-битного слова (IDE контроллер).
outl / inl	Запись / Чтение 32-битного двойного слова (PCI, современные контроллеры).
io_wait	Короткая пауза для стабилизации шины (используется при программировании PIC).
Резюме по глобальным объектам
current_task: Указатель на структуру текущей выполняемой задачи.
next_pid: Счетчик для генерации уникальных идентификаторов процессов.
find_task(pid): Глобальный метод для поиска задачи в кольцевом списке.

7. Управление процессами (Process Management)
7.1 Создание задачи (create_task)
Функция выполняет «ручную» подготовку стека новой задачи, чтобы планировщик мог переключиться на неё так, будто она уже выполнялась ранее.
Аллокация: Выделяется структура task_t и целая страница памяти (4 КБ) под стек через kmalloc_page(). Базовый адрес сохраняется в steck_base для последующей очистки.
Инициализация стека (Stack Framing):
Указатель стека (esp) устанавливается на конец выделенной страницы (стек растет вниз).
В стек записывается «фиктивный» контекст процессора (registers_t):
EFLAGS: Значение 0x202 (биты 1 и 9) гарантирует, что при старте задачи прерывания будут разрешены (IF=1).
CS/EIP: Указывают на сегмент кода ядра и точку входа в функцию задачи (entry_point).
Регистры: Все регистры общего назначения (GPR) и сегментные регистры (DS, ES, FS, GS) инициализируются значениями по умолчанию (0x10 для данных).
Регистрация: Задача получает уникальный PID, имя и вставляется в кольцевой связный список сразу после current_task.
7.2 Завершение задачи (shell_kill_proc)
Функция реализует механизм мягкого завершения процесса по запросу (например, из Shell).
Защита ядра (Kernel Protection): Введен жесткий запрет на завершение процессов с PID 0 и 1 (обычно это само ядро и системный инициализатор). Попытка убить их выведет «Access Denied».
Механизм: Функция не удаляет задачу немедленно (это вызвало бы крах, если задача выполняется сейчас). Вместо этого она переводит статус в TASK_STATE_KILLED (2).
Результат: На следующем тике таймера планировщик увидит эту метку, вырежет задачу из списка и передаст её «Жнецу» для очистки ресурсов.
Справочник по состояниям задач
Состояние	Константа	Описание
RUNNING	0	Задача находится в кольце планировщика и получает кванты времени.
SLEEP	1	(Резерв) Задача ожидает события и пропускается планировщиком.
KILLED	2	Задача помечена как «зомби». Ожидает извлечения из списка и утилизации Жнецом.
Рекомендации по расширению (из кода):
Проверка памяти: В create_task добавлена проверка stack_mem == 0. Это критически важно для предотвращения падения ядра при нехватке RAM.
Безопасность строк: Имя копируется с проверкой длины (до 19 символов) и принудительным терминирующим нулем, что защищает от переполнения буфера в структуре task_t.

8. Драйвер дисковой подсистемы (IDE ATA)
Драйвер управляет основным контроллером жесткого диска, используя порты ввода-вывода (I/O) и механизм прерываний для подтверждения готовности данных.
8.1 Константы и интерфейсы
Базовый порт (0x1F0): Используется для передачи данных (Data Port).
Управляющие порты (0x1F2-0x1F7): Служат для установки номера сектора (LBA), количества секторов и отправки команд (0x20 — чтение, 0x30 — запись, 0xE7 — сброс кэша).
Альтернативный статус (0x3F6): Используется для чтения состояния диска без сброса прерывания.
8.2 Логика ожидания и синхронизации
ide_delay(): Генерирует задержку ~400 нс через чтение порта статуса. Это необходимо для старых устройств, которым нужно время на обновление флагов после команды.
ide_wait_ready(): Опрашивает контроллер до тех пор, пока:
Бит BSY (Busy) не будет сброшен.
Бит DRQ (Data Request) не будет установлен (если требуется передача данных).
Возвращает ошибки, если контроллер отсутствует (0xFF) или диск сообщает о сбое (ERR).
8.3 Чтение сектора (ide_read_sector)
Выбор диска: Устанавливается режим LBA и старшие биты адреса в порту 0x1F6.
Запрос: Отправляется номер сектора и команда 0x20.
Ожидание IRQ: Ядро входит в цикл ожидания флага ide_irq_fired. Реализован timeout, предотвращающий зависание системы при аппаратном сбое.
Передача данных: После подтверждения готовности (прерывание или DRQ) 512 байт считываются из порта данных 0x1F0 в буфер по 2 байта (16-битные слова).
8.4 Запись сектора (ide_write_sector)
Подготовка: Проверка готовности и передача параметров адресации.
Команда: Отправка кода 0x30 (Write Sector).
Передача: Ядро ждет флага DRQ и записывает содержимое буфера в порт данных.
Безопасность данных: В конце вызывается команда 0xE7 (Flush Cache). Это гарантирует, что данные физически записаны на пластины диска, а не остались во внутреннем кэше контроллера.
Взаимосвязь с другими модулями
Прерывания: Функция ide_handler (из второй части кода) напрямую связана с этим драйвером через флаг ide_irq_fired.
Системные вызовы: Сисколл case 4 (read_sector) является оберткой над этим драйвером.
Файловая система: Модуль fat32.h вызывает эти функции для работы с таблицей распределения файлов и чтением каталогов.

9. Драйвер клавиатуры (PS/2 Keyboard)
Драйвер обрабатывает прерывания от контроллера клавиатуры (IRQ 1) и выполняет маппинг скан-кодов в символы.
9.1 Карты символов (Scancode Mapping)
Используются две статические таблицы (массивы на 128 элементов), соответствующие стандартному набору Set 1:
kbd_map: Стандартная раскладка (строчные буквы, цифры).
kbd_map_shift: Раскладка при зажатой клавише Shift (заглавные буквы, спецсимволы).
Спецсимволы: Поддерживаются управляющие символы, такие как \b (Backspace), \n (Enter) и \t (Tab).
9.2 Логика обработки скан-кодов (get_char_from_scancode)
Функция реализует конечный автомат для отслеживания состояния клавиатуры:
Управление модификаторами:
При получении кодов нажатия (Make-коды: 0x2A, 0x36) переменная shift_active устанавливается в 1.
При получении кодов отпускания (Break-коды: 0xAA, 0xB6) — сбрасывается в 0.
Фильтрация:
Игнорируются все Break-коды (у которых установлен 7-й бит), кроме клавиш Shift. Это предотвращает повторный ввод символа при отпускании клавиши.
Добавлена проверка границ массива (scancode >= 128) для предотвращения чтения за пределами памяти (защита от «мусора» на шине данных).
Трансляция: Возвращает символ из соответствующей таблицы в зависимости от флага shift_active.
9.3 Поток данных (Data Flow)
Контроллер: Генерирует IRQ 1.
Exception Handler: Вызывает inb(0x60) для чтения байта данных.
Драйвер: Вызывает get_char_from_scancode.
Потребитель:
Если активен системный вызов чтения (syscall_waiting), символ попадает в key_buffer.
В обычном режиме символ передается напрямую в shell_handle_char для обработки командной строкой.

10. Компоновка ядра (Linker Script)
Сценарий обеспечивает совместимость со спецификацией Multiboot и подготавливает почву для работы менеджера памяти.
10.1 Адресация и точки входа
ENTRY(_start): Определяет глобальную метку _start (обычно находится в boot.asm) как точку входа в программу.
Адрес загрузки (1M): Ядро размещается по адресу 1 МиБ. Это стандарт для Multiboot-ядер, позволяющий избежать конфликтов с зарезервированными областями BIOS и VGA-памяти в первом мегабайте.
10.2 Структура секций
Все секции выровнены по границе 4 КБ (ALIGN(4K)), что критически важно для работы VMM (Virtual Memory Manager), так как управление памятью осуществляется страницами именно такого размера.
.text (Код):
KEEP(*(.multiboot)): Гарантирует, что заголовок Multiboot будет в самом начале файла. Это необходимо, чтобы загрузчик (GRUB) смог распознать ядро.
Содержит исполняемый машинный код.
.rodata (Константы): Хранит неизменяемые данные, такие как строки (например, "Error: Invalid Multiboot Magic!").
.data (Инициализированные переменные): Содержит глобальные переменные, у которых есть начальное значение.
.bss (Неинициализированные переменные):
Хранит глобальные переменные, инициализированные нулем.
*(COMMON): Включает общие символы из разных объектных файлов.
Эта секция не занимает места в файле на диске, но расширяется в памяти при загрузке.
10.3 Маркеры границ
ALIGN(4096): Гарантирует, что конец ядра также выровнен по странице.
_end / end: Эти символы экспортируются и могут быть использованы в Си-коде (например, в pmm_init). Они указывают на первый свободный байт памяти сразу после ядра. Все, что идет после _end, менеджер памяти может считать доступным для кучи (heap) или стека задач.

Используется GNU Make для управления компиляцией и автоматизации создания образов.
11.1 Параметры компиляции
CFLAGS (Си):
-m32: Генерация 32-битного кода (архитектура i386).
-ffreestanding: Указывает, что стандартная библиотека Си отсутствует (ядро само себе библиотека).
-fno-stack-protector: Отключение защиты стека (так как в ядре еще не реализован соответствующий рантайм).
-nostdlib: Полный отказ от стандартных библиотек.
LDFLAGS (Компоновщик):
-T linker.ld: Использование ранее описанного сценария компоновки.
-no-pie: Отключение позиционно-независимого кода (ядро жестко привязано к адресу 1 МБ).

Создает структуру каталогов для GRUB.

12. Менеджмент памяти (PMM и Аллокаторы)
Система управления памятью разделена на физический уровень (блоки) и логический уровень ядра (kmalloc).
12.1 Физический менеджер (PMM)
Использует алгоритм Bitmap (битовая карта) для отслеживания состояния каждого физического кадра памяти (страницы 4 КБ).
Инициализация (pmm_init):
Принимает общий объем памяти, полученный из Multiboot.
Размещает битмап сразу за концом кода ядра (метка &end).
Защита ядра: Все блоки до конца битмапа помечаются как занятые. Свободными объявляются только блоки, идущие после системных данных.
Аллокация (pmm_alloc_block): Ищет первый свободный бит в карте. При нахождении обнуляет содержимое страницы (1024 слова по 4 байта), что критически важно для корректной работы таблиц страниц (VMM).
Множественная аллокация (pmm_alloc_blocks): Позволяет выделять непрерывные участки физической памяти (необходимо для DMA или структур, требующих линейности).
12.2 Виртуальный менеджер и kmalloc_page
Связывает физические кадры с виртуальным адресным пространством.
kmalloc_page: Выделяет физический блок и отображает его на виртуальный адрес, начиная с 0x400000. Это обеспечивает изоляцию и предсказуемость адресов для динамических структур.
kfree_page: Разрывает связь в таблицах страниц (unmap) и возвращает блок в PMM.
12.3 Аллокатор кучи ядра (kmalloc)
Реализует простейший алгоритм Bump Allocator (аллокатор с инкрементным указателем) для мелких структур данных (например, task_t или resource_t).
Логика: Хранит статический указатель current_ptr. При каждом запросе просто сдвигает его на размер size.
Выравнивание: Каждый выделенный блок принудительно выравнивается по 4 байта. Это предотвращает падение производительности и ошибки доступа к памяти (Alignment Faults) на архитектуре x86 при работе со структурами.
Размещение: Куча ядра начинает расти сразу после битмапа и небольшого защитного интервала (4096 байт).
Схема распределения памяти (Memory Map)
Адрес	Описание
0x000000 - 0x100000	Резерв BIOS, VGA, ROM.
0x100000 (1 MB)	Начало кода ядра (секции .text, .data, .bss).
&end	Начало битовой карты PMM.
&end + bitmap_size	Начало динамической кучи ядра (kmalloc).
0x400000	Начало виртуальных адресов для kmalloc_page.

13. Драйвер реального времени (RTC/CMOS)
Драйвер обеспечивает получение текущего времени (часы, минуты, секунды) напрямую из энергонезависимой памяти CMOS.
13.1 Интерфейс CMOS
Взаимодействие происходит через два порта ввода-вывода:
0x70 (CMOS_ADDR): Выбор регистра для чтения/записи.
0x71 (CMOS_DATA): Чтение или запись данных в выбранный регистр.
13.2 Алгоритм чтения времени (rtc_get_time)
Процесс получения данных защищен от ошибок чтения в моменты обновления чипа:
Синхронизация (is_updating): Ядро опрашивает регистр 0x0A. Если 7-й бит установлен, CMOS занят обновлением времени. Драйвер ждет завершения этого процесса, чтобы избежать чтения некорректных (переключающихся) значений.
Чтение регистров: Извлекаются значения из регистров 0x00 (секунды), 0x02 (минуты) и 0x04 (часы).
Декодирование форматов:
BCD в Binary: Большинство систем хранят время в формате BCD (Binary-Coded Decimal). Функция bcd_to_bin преобразует их в обычные целые числа. Проверка формата выполняется через регистр B (бит 2).
12/24 часовой формат: Если система работает в 12-часовом режиме (бит 1 регистра B), драйвер корректирует значение часа, проверяя флаг PM (8-й бит значения часов) и преобразуя его в 24-часовой формат.

14. Командный интерпретатор (Shell)
Shell реализован как интерактивный цикл, обрабатывающий ввод в реальном времени через драйвер клавиатуры.
14.1 Управление вводом (shell_handle_char)
Буферизация: Вводимые символы накапливаются в статическом буфере MAX_BUF (128 байт).
Редактирование: Поддерживается клавиша Backspace (\b), которая корректно уменьшает указатель буфера и обновляет VGA-дисплей.
Исполнение: При нажатии Enter (\n) строка завершается нулем и передается в обработчик shell_execute.
14.2 Группы команд и функциональность
Интерпретатор разделяет команды на логические категории:
А. Информационные и системные
help: Вывод списка доступных команд.
cls: Очистка экрана через vga_clear.
time: Вывод текущего времени (использует RTC драйвер).
mem: Информация о памяти (общее количество страниц и свободные блоки PMM).
Б. Файловая система (FAT32)
ls: Список файлов в корневом каталоге.
cat [file]: Чтение текстового файла (загружает содержимое по адресу 0x200000).
create / delete: Управление файлами.
write [file] [text]: Запись данных. Реализована защита ядра: запрещено удалять или перезаписывать KERNEL.BIN и BOOT.ASM.
В. Управление процессами
list_proc: Выводит таблицу активных задач (PID, имя, статус). Для удобства пользователя статусы заменяются на текст: RUNNING, SLEEP, ZOMBIE.
run [file]: Загрузчик бинарных файлов. Считывает программу по адресу 0x300000 и передает туда управление через указатель на функцию.
kill [pid]: Маркирует процесс для удаления. Включает Critical Protection: нельзя убить процессы с PID < 2 (Ядро и Shell) и процесс-очиститель (cleaner).
14.3 Вспомогательные утилиты (Utils)
Поскольку стандартная библиотека (libc) отсутствует, Shell включает собственные реализации базовых функций:
_strcmp / _strncmp: Сравнение строк.
_atoi: Конвертация строки в число (используется для команды kill).
Резюме архитектуры интерфейса
Компонент	Роль
Buffer Management	Сбор данных до нажатия Enter.
Command Dispatcher	Парсинг строки и вызов функций ядра.
Security Layer	Проверка прав доступа к системным файлам и критическим PID.
Visual Output	Форматирование таблиц процессов и системных данных.

15. Пользовательское приложение (API Example)
Программа скомпилирована в плоский бинарный формат (flat binary) и предназначена для загрузки по адресу 0x300000, что соответствует логике команды run в Shell.
15.1 Использование системных вызовов (Int 0x30)
Приложение взаимодействует с ядром, используя регистр EAX для передачи номера функции и EBX, ECX, EDX для аргументов:
Функция 5 (Create File): Создает файл с именем TEST.TXT.
Функция 7 (Write File): Записывает строку "Hello from ASM App!" в созданный файл. Длина данных вычисляется ассемблером автоматически (data_len).
Функция 1 (Print): Выводит в консоль Shell уведомление о завершении работы и инструкцию для пользователя.
15.2 Структура бинарного файла
Директива org 0x300000: Гарантирует, что все смещения переменных (filename, msg_done) будут рассчитаны правильно относительно адреса загрузки в памяти.
Завершение (ret): Поскольку Shell вызывает программу как функцию (void (*entry)()), инструкция ret корректно возвращает управление обратно в оболочку после выполнения кода.

16. Видеоподсистема (VGA Text Mode Driver)
Драйвер управляет текстовым буфером по адресу 0xB8000 и аппаратными портами видеоконтроллера.
16.1 Управление курсором и прокрутка
Аппаратный курсор (update_cursor): Использует порты 0x3D4 и 0x3D5 для передачи позиции мигающего указателя видеокарте. Это позволяет пользователю видеть место ввода в Shell.
Механизм скроллинга (scroll): При достижении 25-й строки драйвер копирует содержимое экрана на одну строку вверх, освобождая место внизу. Это критически важно для работы длинных выводов (например, команды ls или help).
16.2 Функции вывода данных
vga_putc (Вывод символа):
Обрабатывает управляющие символы: \n (перевод строки) и \b (забой/backspace).
При получении \b драйвер не просто сдвигает курсор, но и заменяет символ в видеопамяти пробелом.
vga_puts: Базовая функция для вывода строковых литералов.
Числовой вывод:
vga_put_int: Преобразование и вывод десятичных чисел (используется для PID, объема памяти и времени).
vga_put_hex: Вывод адресов памяти в шестнадцатеричном формате (незаменимо для отладки указателей и VMM).

17. Виртуальная память (VMM / Paging)
VMM управляет отображением виртуальных адресов на физические, используя двухуровневую структуру: Каталог страниц (Page Directory) и Таблицы страниц (Page Tables).
17.1 Инициализация и Identity Mapping (vmm_init)
Создание каталога: Выделяется физический блок для page_directory.
Identity Mapping (0-4 МБ): Первые 4 мегабайта памяти отображаются «сами на себя» (виртуальный адрес равен физическому). Это необходимо, чтобы после включения пейджинга ядро не потеряло доступ к своему коду, видеопамяти (0xB8000) и структурам Multiboot.
Активация: Физический адрес каталога загружается в регистр CR3, после чего в регистре CR0 устанавливается 31-й бит (PG), что переводит процессор в режим страничной адресации.
17.2 Механизм отображения (vmm_map)
Функция связывает виртуальный адрес с физическим кадром:
Индексация: Виртуальный адрес делится на pd_index (старшие 10 бит) и pt_index (средние 10 бит).
Динамическое создание таблиц: Если для данного участка памяти таблица страниц еще не создана (PAGE_PRESENT отсутствует), VMM запрашивает у физического менеджера (PMM) новый блок и регистрирует его в каталоге.
Флаги доступа:
PAGE_PRESENT (0x1) — страница в памяти.
PAGE_WRITE (0x2) — разрешена запись.
PAGE_USER (0x4) — доступ для пользовательских программ (User Mode).
Инвалидация TLB: Команда invlpg сбрасывает кэш ассоциативной трансляции для данного адреса, чтобы изменения вступили в силу немедленно.
17.3 Освобождение памяти (vmm_unmap)
Обеспечивает корректное «разрывание» связи между адресами:
Находит физический адрес (фрейм) через таблицу страниц.
Возврат ресурсов: Вызывает pmm_free_block, помечая физическую память как свободную для других задач.
Очистка: Зануляет запись в таблице страниц и обновляет кэш TLB.
Техническое резюме подсистемы памяти
Уровень	Модуль	Роль
Физический (PMM)	pmm.c	Управляет «железом» (битами в Bitmap), выделяет страницы по 4КБ.
Виртуальный (VMM)	vmm.c	Управляет таблицами страниц, изолирует память процессов, включает PG-бит.
Логический (Heap)	kmalloc	Выделяет мелкие куски памяти внутри ядра (структуры, строки).

1. Архитектура и Инициализация
MBR-Aware: Драйвер не просто ищет ФС с нуля, а парсит таблицу разделов (Master Boot Record), находит активный раздел FAT32 и вычисляет его partition_lba_start.
BPB Parsing: Извлекает геометрию (резервные сектора, количество таблиц FAT, размер кластера).
Cluster Mapping: Реализована трансляция логических кластеров в физические LBA-сектора с учетом смещения области данных.
2. Управление файлами (CRUD)
Create (Создание): Поиск свободных слотов (0x00 или 0xE5) в директории и первичная аллокация кластера.
Read (Чтение): Потоковое чтение всей цепочки кластеров по таблице FAT до маркера EOF.
Update (Запись/Обновление): Динамическое расширение файлов. Если данные не влезают, драйвер на лету находит свободные кластеры, линкует их в цепочку и обновляет размер в дескрипторе.
Delete (Удаление): Стирание данных (заполнение нулями), освобождение цепочки в FAT и маркировка записи байтом 0xE5.
3. Критические механизмы
8.3 Format: Автоматическая конвертация имен (например, test.txt -> TEST TXT) с приведением к верхнему регистру.
Синхронизация: Использование cli/sti при записи в таблицы FAT, чтобы прерывание таймера (переключение задач) не «развалило» файловую систему в момент правки указателей.
Безопасность: Проверка имен файлов для защиты критических системных компонентов (KERNEL.BIN, BOOT.ASM) от удаления через Shell.
